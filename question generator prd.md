## 1. Executive Summary
The Question Generator is the core authoring engine of the assessment platform. It converts
source material (typically a PDF) into structured, schema-compliant assessment questions
across seven supported question types, and allows generated question sets to be exported as a
portable JSON file.
This revision of the PRD addresses two production-blocking defects and introduces one new
capability:
- Per-type question count enforcement — the generator currently ignores user-selected
counts per question type and instead produces a fixed default volume (~20 questions),
regardless of input.
- Strict JSON export — there is currently no way to download a generated question set in
a predictable, schema-validated format.
- Role-aware access — generation, review, and export behavior must now be
differentiated across four platform roles: Principal, HOD, Teacher, and Student.
The intended outcome is a generator that is deterministic (exact counts, no silent defaults),
trustworthy (every export is schema-valid), and appropriately scoped per role (Students never
generate or export; Teachers author; HODs and Principals oversee).
## 2. Problem Statement & Goals
## 2.1 Problem Statement
Teachers configure exact question counts per type (for example, 10 Fill in the Blanks, 10 Match
the Following, 5 MCQ) expecting a total of 25 questions. The system instead silently falls back
to a fixed/default generation volume of approximately 20 questions, ignoring the per-type counts
entirely. This breaks lesson planning, grading weightage, and time-boxed assessments built
around a specific question count.
In parallel, there is no supported way to extract a generated question set out of the platform.
Teachers and HODs who need to archive, share, or feed question sets into external tooling
currently have no export path, and any ad-hoc workaround risks producing malformed or
incomplete data.
Finally, today's generator has no concept of role. Every authenticated user effectively has the
same generation and export capabilities, which is inappropriate once Students are given
visibility into the platform — Students must never be able to trigger generation, view answer
keys prior to release, or export raw question banks.
## 2.2 Goals
-  Generate the exact number of questions per type, every time, with zero default-count
interference.
-  Guarantee that every exported JSON file is schema-valid and immediately usable
without manual correction.

-  Introduce four distinct roles — Principal, HOD, Teacher, Student — each with clearly

## 3. Users
The platform recognizes four roles. Permissions are strictly additive top-down for oversight
(Principal sees everything HOD sees, and more) but generation and export remain Teacher-
exclusive actions — seniority does not grant authoring capability by default.
RoleDescription
## Principal
Institution-level oversight. Views generation activity, question quality metrics, and
export logs across all departments and teachers. Does not generate or edit
questions directly.
## HOD
Department-level oversight. Reviews and approves question sets generated by
teachers within their department before they are published to students. Can
request regeneration of a type. Does not generate from scratch.
## Teacher
Primary author. Configures source content, selects question types and per-type
counts, triggers generation, edits/reviews generated questions, and exports the
final JSON file.
## Student
Consumer of approved assessments only. No access to generation, export, raw
JSON, or answer keys prior to assessment release. Sees only the rendered,
approved question set when assigned.

## 5. Feature Requirements
5.1 Feature 1 — Per-Type Question Count Fix
## Problem
The system currently generates a fixed/default number of questions (approximately 20), ignoring
user-selected counts per question type.
## Expected Behavior
A Teacher selects a count per type. For example:
•Fill in the Blanks:10

•Match the Following:10
## •MCQ:5
## Functional Requirements
-  Each question type is processed independently of every other type.
-  Generation strictly follows the count provided per type; there is no shared budget across
types.
-  No global or default count exists anywhere in the system — every count is explicit and
## Teacher-supplied.
-  When multiple types are selected, each is generated separately and the results are
combined into a single set.
## Validation Rules
-  Total questions returned must equal the sum of all selected per-type counts.
•If generation returns more than requested:the excess is trimmed before being
returned.
•If generation returns fewer than requested:the system retries generation for the
shortfall, or fails with an explicit error for that type if retries are exhausted.
-  Every question must strictly follow its type's schema.
-  Every question id must be unique across the full combined dataset, not just within its
own type.
## Edge Cases
•Count = 0:that type is ignored entirely — not generated, not requested, not present in
output.
•Insufficient source content:if the source PDF cannot support the requested count for
a type, the system shows a failure message scoped to that specific type, without
blocking the types that did succeed.
-  The system actively prevents duplicate questions both within a type and across different
types.
5.2 Feature 2 — Export Questions (JSON Download)
## Goal
Allow Teachers to download a generated question set as a single .json file in a strict, predefined
format.
## User Flow
- Teacher generates questions for one or more types.
- The “Export Questions” button becomes visible once generation succeeds.
- Teacher clicks Export.
- The system downloads a JSON file directly to the device.
## Functional Requirements

-  The export includes all generated questions across all generated types.
-  Only types that were actually generated are included; unselected or zero-count types
never appear in the file.
-  File name format: questions_<timestamp>.json
-  Export is a direct, automatic download — no manual copy-paste step.
-  Export is available only to the Teacher role; HOD and Principal may view export history
but not trigger a raw export themselves, and Students cannot access export at all (see
## Section 3.1).
5.3 Role-Based Workflow Requirements
-  Generation and export actions are gated server-side by role, not only hidden client-side.
-  HOD approval is required before a Teacher-generated set is published to Students,
unless an institution-level setting allows Teachers to self-publish.
-  Principal-level views are read-only aggregates and must never expose answer keys or
raw export files directly.
-  Students only ever see the rendered, approved assessment — never the underlying
JSON, never pre-approval drafts, never answer keys before submission.

## 6.3 Common Rules Across All Types
-  id must be unique globally across the entire exported dataset, not just within one type.
-  marks must be numeric on every question.
◦  hide_text
◦  text
◦  read_text
◦  image
-  explanation is mandatory on every question, for every type.
-  No extra fields are allowed beyond what is defined in the schema for that type.
## 6.4 Export Validation Rules
Before allowing a download, the system must:
-  Ensure at least one question exists in the set being exported.
-  Ensure every question in every type follows its schema exactly.
-  Ensure totalMarks for each type block equals the sum of marks across that type's
questions.
◦  Export is blocked — no partial or best-effort file is produced.
◦  The system shows the error: “Invalid question structure detected.”

7.1 Teacher — Generate and Export
- Teacher uploads or selects a source PDF.
- Teacher selects one or more question types and enters a target count for each (0 to
exclude a type).
- Teacher triggers generation.
- System generates each selected type independently and reports per-type success or
failure.
- On success, Teacher reviews the combined question set inline.
- Teacher optionally edits individual questions or requests regeneration of a single type.
- “Export Questions” becomes visible once at least one type has succeeded.
- Teacher clicks Export; the system validates the full set and downloads
questions_<timestamp>.json, or shows “Invalid question structure detected” if validation
fails.
7.2 HOD — Review and Approve
- HOD opens a department dashboard listing question sets generated by their Teachers.
- HOD opens a set and reviews questions by type, including per-type counts and any
failed types.
- HOD either approves the set for publishing to Students, or sends a regeneration request
back to the Teacher for a specific type.
## 7.3 Principal — Oversight
- Principal opens an institution-wide analytics view showing generation volume, approval
rates, and export activity across departments.
- Principal can drill into a department to see aggregate metrics, but cannot open raw
question content or export files directly.
## 7.4 Student — Assessment Access
- Student sees only assessments that an HOD (or self-publishing Teacher, per institution
setting) has approved.
- Student answers the rendered assessment; no raw JSON, draft set, or answer key is
ever exposed before submission.
- Non-Functional Requirements
•Determinism:identical inputs (source content, types, counts) must not silently produce
a different total question count between runs; deviations must surface as explicit per-
type errors, not silent substitution.
•Authorization:generation and export endpoints enforce role checks server-side; client-
side hiding of buttons is not sufficient.
•Auditability:every generation run and export action is logged with role, timestamp, and
per-type outcome, for HOD and Principal visibility.

•Performance:per-type generation failures must not block or delay the return of types
that succeeded.
•Data integrity:exported files must be valid, parseable JSON with no trailing or
malformed content, on every export.
•Backward compatibility:existing question sets generated under the previous default-
count behavior must remain viewable, even though they will not retroactively gain
enforced per-type counts.
## 9. Success Metrics
-  100% of generation requests return exactly the requested count for every selected type,
or an explicit per-type failure — zero silent default-count substitutions.
-  100% of successful exports open without errors and pass schema validation on first
attempt.
-  Zero instances of Students accessing generation, export, or pre-approval question
content in audit logs.
-  Reduction in support tickets related to “wrong number of questions generated” to zero
post-launch.
-  HOD median time-to-approve a submitted question set, tracked as a baseline for future
workflow optimization.
